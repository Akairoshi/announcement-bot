# ADR-002: Clean Architecture + CQRS

- **Статус:** Принято
- **Дата:** 2025-05-17

---

## Контекст

Проект — Telegram-бот для рассылки объявлений по категориям. Разрабатывается в рамках производственной практики с целью демонстрации навыков проектирования и последующего добавления в портфолио.

Требования к архитектуре:
- Чёткое разделение ответственности между слоями
- Возможность тестировать бизнес-логику изолированно
- Читаемая структура для портфолио и код-ревью

---

## Решение

Использовать **Clean Architecture** в связке с **CQRS через MediatR**.

### Структура слоёв

```
Domain          — сущности, перечисления, интерфейсы репозиториев
Application     — бизнес-логика, команды и запросы (CQRS), DTOs
Infrastructure  — EF Core, PostgreSQL, репозитории, фоновые сервисы
Presentation    — обработчики Telegram Update, middleware, точка входа
```

Зависимости направлены только внутрь: `Presentation → Application → Domain`. `Infrastructure` реализует интерфейсы из `Domain` и подключается через DI.

### CQRS

Операции разделены на:
- **Command** — изменяет состояние (создать объявление, подписаться на категорию)
- **Query** — читает состояние (список категорий, профиль пользователя)

Диспетчеризация через `MediatR` — `Application` не знает об инфраструктуре, `Presentation` не знает о деталях бизнес-логики.

---

## Альтернативы

### Простой сервисный слой без CQRS

Один `AnnouncementService` со всеми методами. Проще в начале, но быстро превращается в God Object при росте числа команд. Сложнее тестировать отдельные операции.

### Минимальная архитектура (всё в одном проекте)

Подходит для pet-проектов на 2-3 команды. Для демонстрации навыков на практике и в портфолио — не показательно.

---

## Последствия

- Каждая команда/запрос — отдельный класс, легко найти и изолированно протестировать
- Добавление новой функции не затрагивает существующие обработчики
- Порог входа выше, чем у простого сервисного слоя — оправдан целями проекта
- При необходимости Infrastructure можно заменить (например, сменить БД) без изменения бизнес-логики
