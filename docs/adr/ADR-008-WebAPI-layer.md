# ADR-008: Создание единого WebAPI для интеграций и панели управления

* **Статус:** Принято
* **Дата:** 2026-06-16

---

## Контекст

С ростом проекта `AnnouncementBot` возникла необходимость расширить его возможности в двух направлениях:

1. **Автоматизация (B2B-интеграции):** Предоставить сторонним организациям возможность автоматически создавать объявления в боте из своих внешних систем (CRM, сайты, сервисы мониторинга).
2. **Панель управления (Admin Dashboard):** Создать веб-интерфейс (например, на React) для удобного управления категориями, шаблонами, просмотра расширенной статистики доставки (`DeliveryStatuses`) и логов аудита (`AuditLogs`), так как управление сложными структурами через интерфейс Telegram имеет ограничения по UX.

Перед нами встали архитектурные вызовы:

* Делить ли эти задачи на разные микросервисы?
* Как организовать удобный и безопасный контракт для внешних систем, не заставляя их знать внутренние `Guid` идентификаторы базы данных?
* Как организовать авторизацию в админ-панели без использования громоздкого Telegram Login Widget?

---

## Решение

### 1. Архитектурная структура

Создать один общий проект **`AnnouncementBot.WebApi`** в рамках текущего Solution. Проект будет выступать в роли единого REST API шлюза, но разделять потоки на уровне эндпоинтов и механизмов авторизации:

* Группа `/api/v1/external/*` — для автоматизации внешних систем.
* Группа `/api/v1/admin/*` — для взаимодействия с фронтендом админ-панели.

Бизнес-логика не дублируется: контроллеры остаются "тонкими" и просто транслируют входящие запросы в существующий слой `Application` через MediatR `Command` и `Query`.

### 2. Внешняя автоматизация (Интеграции)

* **Идентификация категорий:** Внешние системы передают категорию текстом (`categoryName`). Обработчик команды на нашей стороне сам ищет уникальное имя категории в БД. Если категория не найдена, возвращается ошибка `404 Not Found` (Вариант с созданием категорий на лету отклонен ради безопасности данных).
* **Поле автора:** Так как у внешней системы нет Telegram User ID, в поле `CreatedById` объявлений записывается ID `SuperAdmin`'а из конфигурационного файла приложения.
* **Безопасность:** Доступ защищается по упрощенной схеме **API-Key Auth**. Ключ передается в HTTP-заголовке `X-Api-Key` и валидируется через кастомный Middleware/Action Filter на основе конфигурации `appsettings.json`.

### 3. Авторизация в Админ-панели (Auth-by-Code)

Вместо интеграции Telegram Login Widget на фронтенд, реализуется двухфакторный сценарий авторизации через самого бота:

1. Пользователь вводит свой Telegram ID на сайте админки.
2. WebAPI проверяет наличие пользователя в БД и его роль (`Admin`/`SuperAdmin`).
3. Генерируется случайный 6-значный одноразовый код со сроком жизни 5 минут (кэшируется в `IMemoryCache`).
4. Бот отправляет этот код пользователю в личные сообщения в Telegram.
5. Пользователь вводит код на сайте, WebAPI сверяет его и генерирует стандартный **JWT-токен** со сроком жизни, содержащий Claims (Id, Username, Role).

## Альтернативы

### Разделение на два отдельных проекта API

* **Плюсы:** Полная изоляция интеграционного шлюза от админки.
* **Минусы:** Избыточное усложнение на данном этапе разработки (Overengineering). Потребуется настраивать разные порты, дублировать подключение к DBContext/UnitOfWork или выносить инфраструктурный слой в отдельные библиотеки, что замедлит разработку.

### Использование Telegram Login Widget для админки

* **Плюсы:** Стандартное решение от Telegram.
* **Минусы:** Требует сложной настройки скриптов на фронтенде, привязки домена к конкретному боту через BotFather, что усложняет локальное тестирование (localhost) во время разработки. Схема Auth-by-Code полностью контролируется нашим бэкендом.

---

## Последствия

* В решении появляется новый слой/проект `AnnouncementBot.WebApi`.
* В слое `Application` добавляются новые команды (например, `CreateExternalAnnouncementCommand`).
* Настройка CORS в WebAPI для безопасного взаимодействия с будущим фронтендом на React.
* Появляется зависимость от пакета `Microsoft.AspNetCore.Authentication.JwtBearer`.
* Появляется необходимость использования `IMemoryCache` (или распределенного кэша) для хранения временных кодов авторизации.
