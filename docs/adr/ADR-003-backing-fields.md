# ADR-003: Backing Fields для навигационных свойств

- **Статус:** Принято
- **Дата:** 2025-05-18

---

## Контекст

Доменные сущности инкапсулируют своё состояние — навигационные коллекции объявлены как `IReadOnlyCollection<T>` с приватным `List<T>` внутри. Это защищает коллекции от модификации снаружи.

EF Core при материализации объектов из БД должен заполнять эти коллекции. По умолчанию он ищет публичный сеттер или публичную коллекцию — их нет.

---

## Решение

Использовать **Backing Fields** — явно указывать EF Core на приватное поле через конфигурацию:

```csharp
// Domain/Entities/User.cs
private readonly List<Subscription> _subscriptions = [];
public IReadOnlyCollection<Subscription> Subscriptions => _subscriptions.AsReadOnly();

// Infrastructure/Persistence/Configurations/UserConfiguration.cs
builder.Navigation(u => u.Subscriptions).HasField("_subscriptions");
```

EF Core получает доступ к приватному полю через рефлексию и заполняет его напрямую, минуя публичный интерфейс.

---

## Альтернативы

### Публичный `ICollection<T>` с `private set`

```csharp
public ICollection<Subscription> Subscriptions { get; private set; } = [];
```

EF Core работает без дополнительной настройки. Но коллекцию можно модифицировать через `.Add()` и `.Remove()` снаружи — инкапсуляция нарушена.

### Отказ от навигационных свойств в Domain

Убрать коллекции из сущностей, загружать связанные данные только через явные запросы в репозиториях. Чистый Domain, но теряется удобство EF Core — каждая связь требует отдельного запроса или ручного джойна.

---

## Последствия

- Инкапсуляция сохранена — снаружи коллекция только для чтения
- Каждая навигационная коллекция требует явной настройки `HasField` в конфигурации
- EF Core использует рефлексию для доступа к приватному полю — это стандартный поддерживаемый сценарий, не хак
